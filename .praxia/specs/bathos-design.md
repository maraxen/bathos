# bathos — Design Spec
**Date:** 2026-05-15  
**Status:** Approved  
**CLI command:** `bth`

---

## Problem

A single researcher running experiments across 10+ independent git repos and a SLURM cluster loses track of:
1. What ran — command, args, git commit, environment
2. Where results live — output files, parquet/CSV
3. Whether results are still valid — code has changed since the run

Existing tools (MLflow, DVC, W&B, Sacred, Aim) each cover at most two of the three failure modes, and none are SLURM-safe for parallel job arrays without a running server.

## Non-Goals

- Not a hyperparameter sweep framework (use Optuna / Ray Tune)
- Not a model registry (use MLflow for that)
- Not a pipeline orchestrator (use DVC / Prefect)
- Not a team collaboration tool — single-researcher design center
- No cloud sync, no SaaS account required

---

## Architecture

### Catalog

Central catalog at `~/.bth/catalog/` using **DuckDB + Parquet**. No server. No daemon.

- `~/.bth/catalog/runs.parquet` — append-only run records
- `~/.bth/catalog/outputs.parquet` — output file registrations, FK to run id
- DuckDB opened read/write inline per command; SLURM-safe via write-ahead append pattern

Cross-project namespacing via `project_slug` column; all projects write to the same catalog files.

### Project Registration

Each research project gets a `.bth.toml` at its root (created by `bth init`):

```toml
[project]
slug = "prolix"
root = "/home/marielle/projects/prolix"
```

### Run Record Schema

| Field | Type | Description |
|---|---|---|
| `id` | UUID | Run identifier |
| `project_slug` | str | From `.bth.toml` |
| `command` | str | Full command string |
| `argv` | str[] | Parsed argument list |
| `git_hash` | str | HEAD SHA at run time |
| `git_branch` | str | Branch name |
| `git_dirty` | bool | Uncommitted changes present |
| `timestamp` | datetime | UTC start time |
| `duration_s` | float | Wall time |
| `exit_code` | int | Process exit code |
| `status` | enum | `running` / `completed` / `failed` / `killed` |
| `output_paths` | str[] | Registered output files |
| `tags` | str[] | Optional user-supplied tags |

## CLI

```
bth init                            # register project, create .bth.toml + scripts/ skeleton
bth run <script> [-- <args>]        # wrap any script; capture provenance automatically
bth ls [--project P] [--since Nd]   # list runs
bth show <run-id>                   # full run detail
bth find --since 7d --status failed # filtered query
bth check [output-file]             # validity: is recorded git_hash == HEAD?
bth sql "<duckdb sql>"              # escape hatch: raw DuckDB query against catalog
```

`bth run` pre-run: captures git state, writes a `running` record.  
`bth run` post-run: updates status, duration, exit code; registers output files if `--out` was passed through to the script.

## Python Decorator

For Typer-based scripts that integrate directly:

```python
import bth

@bth.experiment(name="benchmark_tip3p", tags=["tip3p", "nvt"])
def main(n_steps: int, out: Path):
    ...  # results written to out
```

The decorator captures the same provenance fields as the CLI wrapper. Optional Trackio delegation for in-process metric streaming (see Phase 2+).

## Script Directory Convention

Applied via `bth init` scaffold and documented in each project's CLAUDE.md:

| Directory | Purpose | Naming | bth-tracked |
|---|---|---|---|
| `scripts/experiments/` | Typer experiment runners | `verb_noun.py` | Yes |
| `scripts/analysis/` | Post-hoc analysis, plots | `verb_noun.py` | Optional |
| `scripts/validation/` | Correctness checks | `verb_noun.py` | Optional |
| `scripts/benchmarks/` | Performance benchmarks | `verb_noun.py` | Yes |
| `scripts/data/` | Data pipeline / conversion | `verb_noun.py` | No |
| `scripts/slurm/` | SLURM `.slurm` + env helpers | `verb_noun.slurm` | Via wrapper |
| `scripts/debug/` | Debug specific issues | `YYMMDD_desc.py` | No |
| `scripts/explore/` | Open-ended investigation | `YYMMDD_desc.py` | No |
| `scripts/scratch/` | Catchall / ephemeral | `YYMMDD_desc.py` | No (gitignored) |

## SLURM Integration

Each project's `scripts/slurm/_bth_env.sh` (generated by `bth init`) sources common env and sets the `BTH_PROJECT_SLUG` env var so job wrappers can call `bth run` transparently from within a SLURM job.

SLURM-safety: DuckDB Parquet appends are atomic at the file level via write-then-rename. Parallel job arrays do not lock each other.

## Storage Layout

```
~/.bth/
  catalog/
    runs.parquet
    outputs.parquet
  config.toml          # global bth config (optional overrides)
```

Per-project:
```
<project-root>/
  .bth.toml            # slug + root
  scripts/
    experiments/
    analysis/
    ...
    scratch/           # .gitignore entry
```
