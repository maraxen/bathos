# Brainstorm: Bathos claim-tier rigor: open design calls for campaign-level hypothesis slate, discriminability map authoring strategy, union gate strictness policy at MVP, claim.bth.toml placement and schema, and scoping of adjacent concerns (baseline admissibility §10.1, signal discrimination §10.3) — producing a complete implementable spec for the bathos backlog DAG

## Problem Frame
Fixed constraints (non-negotiable):
1. Language-agnostic sidecar design — `.bth.toml` sidecars must be plain TOML parseable without importing Python; no runtime evaluation, no UUIDs that don't exist until `bth run` executes.
2. Single-researcher context on SLURM — tooling must work in batch jobs where the researcher is not present to answer prompts; hard gates must be survivable at submission time, not only at run time.
3. Existing DuckDB+Parquet schema is additive-only — no breaking migrations; new fields (`claim_path`, `claim_sha256`, `claim_discriminates`, `claim_isolates`) must be nullable columns added via `bth migrate`.
4. Postmortem scaffold/validate pattern is the proven model — any claim-tier tooling must follow the same scaffold() + validate() shape already proven in `postmortem.py` and exposed via MCP.
5. Adoption ladder: Voluntary → Expected → Required — MVP must not mandate claim authoring for all campaigns; it gates on specific campaign modes or explicit opt-in.
6. `outcome_label` is already a free string in the Campaign dataclass — `confounded` verdict is additive, not a new status enum value.

Negotiable:
- Whether discriminability checking is enforcement-time (lint, submit gate) vs query-time (report generation)
- Whether `claim.bth.toml` lives in project tree, catalog sidecar dir, or both
- Whether `discriminates`/`isolates` are sidecar fields, run-record fields, or CLI flags
- Whether §10.1 baseline_parity and §10.3 probe vocabulary live inside bathos enforcement vs skill-only methodology
- Union Gate escalation policy shape at MVP

The success condition is: after this session, each of the five questions has a single chosen answer with enough detail that a developer can open a PR without a follow-up design meeting.

## Idea Pool
- [user] PEGS decomposition:
- [user] Processes: claim authoring (researcher writes claim.bth.toml), campaign conclusion (bth conclude sets outcome_label), union gate check (validates discriminability before verdict is written), lint (design-time static checks), submit gate (enforcement at bth submit), MCP report generation, migration (bth migrate adds columns).
- [user] Events: researcher starts a new confirmation campaign, researcher authors a claim file, bth run completes and writes run record, bth conclude is called, bth lint runs in CI, audit review of a concluded campaign.
- [user] Goals: prevent EPIC-6-class errors (outcome label ambiguity between competing hypotheses), make claim authoring low-ceremony for a single researcher, produce auditable campaign records, keep SLURM jobs unblocked, remain language-agnostic.
- [user] States: campaign mode (exploration/calibration/confirmation/validation/production), claim_path NULL vs populated, outcome_label set vs unset, union gate: passed / warned / confounded / blocked.
- [user] Key assumptions to surface:
- [user] Discriminability matrix is authored before runs complete (pre-registration constraint)
- [user] A researcher can name outcome labels before they run experiments (this is the same assumption behind existing sidecar hypothesis authoring)
- [user] `confounded` as a verdict state is displayable by existing rich_fmt formatters (it is — free string)
- [user] §10.3 probe vocabulary is purely methodological — bathos cannot validate quality of scientific probing, only structural presence of calibration runs
- [ai] Q1-A: Hand-authored discriminability matrix in claim.bth.toml as a TOML inline table — [[discriminability]] rows with keys `outcome_a`, `outcome_b`, `verdict` (discriminates/confounds/indeterminate). Researcher uses the same outcome-label vocabulary they already use in sidecar [outcomes] blocks. No new vocabulary to learn; validated at lint time against actual run outcome_labels in the catalog.
- [ai] Q1-B: Auto-derive discriminability from [[hypotheses]].predicted_signature blocks — each hypothesis declares expected metric fingerprint; the system computes pairwise overlap to determine confoundability. Eliminates hand-authoring burden but requires a predicted_signature schema that doesn't exist yet, and requires metric-space reasoning that bathos cannot validate without domain knowledge.
- [ai] Q1-C: Lint-only discriminability — no matrix authored upfront; instead, lint checks post-hoc that runs with the same outcome_label span at least two hypotheses and flags those as potentially confounded. This is detection after the fact, not prevention. Lower ceremony but misses the pre-registration value entirely — the researcher learns about confounding after running, not before designing.
- [ai] Q1-D: Assumption reversal — what if discriminability is not a matrix but a flat list of `[discriminability.pairs]` entries, each naming two outcome labels and asserting `discriminates = true/false`? This is the minimal encoding: no matrix algebra, directly assertable, lint-verifiable against catalog. The researcher only enumerates the pairs they care about, not the full cross-product.
- [ai] Q2-A: Union Gate as warning-only at MVP — `bth conclude` prints a warning if discriminability check fails but writes verdict anyway. Zero friction for adoption. Risk: warning fatigue; researcher learns to ignore the gate. Does not prevent EPIC-6 errors, only annotates them.
- [ai] Q2-B: Union Gate as hard-block at MVP — `bth conclude` exits nonzero if discriminability check fails. Prevents EPIC-6 errors completely. Problem: a researcher mid-SLURM campaign who forgot to author a claim file cannot conclude their campaign without manual intervention; unacceptable for SLURM batch context where researcher may be away.
- [ai] Q2-C: Union Gate as soft-block — `bth conclude` succeeds but writes `outcome_label="confounded"` instead of the researcher-provided label when the discriminability check fails. Verdict is recoverable: researcher authors or fixes the claim file, re-runs `bth conclude --reopen`, and the verdict updates. This fits the existing `outcome_label` free-string model and the adoption ladder: gate exists, but is not catastrophic.
- [ai] Q2-D: Mode-gated escalating policy — Union Gate behavior depends on campaign mode: exploration/calibration → warning-only; confirmation → soft-block (confounded verdict); validation/production → hard-block. This mirrors how `bth submit` already treats reproduction prerequisites differently by mode. Researcher intent (mode) determines strictness; no surprise blocks during early-stage work.
- [ai] Q3-A: Project-tree placement only — `.bth/claims/<name>.claim.toml` in the git-tracked project directory. Claim files are authored, versioned, and reviewed like code. Natural home for pre-registration artifacts. Downside: requires knowing the claim file path at `bth conclude` time; researcher must pass `--claim` flag or convention-based auto-discovery.
- [ai] Q3-B: Catalog sidecar placement only — `.bth/catalog/sidecars/<campaign_id>/claim.toml` written by `bth claim scaffold`. Consistent with existing `emit_campaign_report()` and `emit_figure_manifest()` pattern. Not git-tracked by default (catalog dir is typically gitignored). Loses pre-registration provenance — the researcher can author the claim after seeing results.
- [ai] Q3-C: Two-location model — claim is authored in project tree (`.bth/claims/<name>.claim.toml`, git-tracked), then at `bth conclude` time the claim is copied/linked into the catalog sidecar dir and a SHA256 is recorded in the campaigns table. This gives both git provenance (project-tree copy) and catalog colocation (sidecar copy) at the cost of two files that must stay in sync. The SHA256 is the integrity check: if catalog copy differs from project-tree copy, lint flags it.
- [ai] Q3-D: Assumption reversal — what if the claim file is not a separate file at all, but a `[claim]` block appended to the campaign's existing sidecar (if campaigns have a sidecar), or a `[claim]` block in a new `<campaign-slug>.bth.toml` in the project root? This avoids creating a new directory (`.bth/claims/`) and reuses the existing TOML extension pattern. Risk: campaigns don't currently have a single sidecar file — their data lives in the DuckDB campaigns table, not a TOML.
- [ai] Q4-A: `discriminates`/`isolates` as sidecar fields using researcher-chosen short IDs — the sidecar for a run includes `discriminates = ["H1", "H0_misspec"]` where "H1" and "H0_misspec" are the same short-ID strings used as keys in the claim.bth.toml [[hypotheses]] array. Fully language-agnostic (plain TOML string array), stable at authoring time, matches the label vocabulary in the discriminability matrix.
- [ai] Q4-B: `--discriminates` CLI flag at `bth run` time — researcher passes `bth run script.py --discriminates H1,H0_misspec` at job submission. Values stored in run record (`claim_discriminates` column), never in the sidecar file. Sidecar remains language-agnostic and unchanged. Downside: breaks pre-registration discipline — the intent is not declared in the git-tracked sidecar, only in the CLI invocation which may not be version-controlled. Also breaks SLURM reproducibility: the sbatch script must be updated to include the flag.
- [ai] Q4-C: Campaign UUID as the discriminability key — runs reference their campaign_id (UUID) and the discriminability matrix keys are campaign UUIDs. Eliminates the short-ID vocabulary problem. Fatal flaw: campaign UUIDs are not known when writing the sidecar for a new script (the sidecar is authored before `bth campaign start` is called). Violates the language-agnostic / stable-at-authoring-time constraint.
- [ai] Q4-D: Hybrid — sidecar declares `discriminates` as short IDs (language-agnostic, pre-registration), and `bth run` also accepts `--discriminates` as an override/supplement at submission time. The run record stores both, with the sidecar value taking precedence for pre-registration purposes. CLI flag is for cases where the researcher uses a generic script sidecar across multiple campaigns with different hypothesis slates.
- [ai] Q5-A: `[baseline_parity]` block in bathos sidecar, validated by existing `check_baseline_ref_exists` extension in linter.py — adds `parity_metric`, `parity_tolerance`, `parity_direction` fields to the sidecar, lint validates that `baseline_ref` run exists AND that the referenced run's metric matches within tolerance. This is a natural Tier-2 lint extension: requires one DuckDB query against existing warm catalog. /jax-port handles code-level porting; bathos owns the experimental record of parity.
- [ai] Q5-B: §10.3 signal discrimination/probe vocabulary — bathos enforces structural presence of Stage 0 calibration runs (`stage_name="calibration"` gate already exists via ControlsBlock), but does NOT validate probe design quality. The `/using-bathos` skill documents probe vocabulary (positive controls, negative controls, sensitivity probes) as methodology guidance. Bathos only asks: "does a calibration-stage run exist for this campaign?" not "is your probe scientifically adequate?" This is the correct division of labor.
- [ai] Q5-C: §10.1 in skill-only — baseline admissibility is entirely documented in /using-bathos as methodology; bathos does not add a `[baseline_parity]` block or any new lint checks. Risk: the researcher who was burned by EPIC-6 has no automated check that their port achieved parity — the entire point of the §10 section is machine-verifiable evidence. Skill-only is too weak for this use case.
- [ai] Cross-cutting: `bth claim scaffold` command (mirrors `bth postmortem scaffold`) — takes `--campaign-id` or `--campaign-name`, produces a pre-populated `claim.bth.toml` template in `.bth/claims/` with the campaign's hypothesis already pulled from the campaigns table, outcome_labels enumerated from existing runs, and a discriminability pair list stub. Researcher fills in the `discriminates = true/false` verdicts. MCP tool `claim_scaffold(campaign_id)` exposes the same operation.
- [ai] Cross-cutting: claim SHA256 integrity check at `bth conclude` — when concluding a campaign, bathos reads the claim file, hashes it, and stores the hash in `campaigns.claim_sha256`. If `bth lint` later detects that the claim file on disk has a different SHA256 than the one recorded at conclude-time, it raises a Tier-1 ERROR. This provides tamper-evidence without requiring a separate signing mechanism. Compatible with the postmortem pattern (postmortem_validate already does structural validation; claim adds content integrity).
- [ai] Cross-cutting: `bth claim validate` command — structural validation of a claim.bth.toml against the schema (required fields, type checks, outcome-label cross-reference against campaign runs in catalog). MCP tool `claim_validate(path)`. This is the exact mirror of `bth postmortem validate` and enables pre-commit hooks and CI checks without requiring `bth conclude` to run first.
- [ai] Assumption reversal on Q2: What if the Union Gate is not at `bth conclude` at all, but is instead a Signal in `bth sprint-audit`? No gate at conclude-time; instead, campaigns with `outcome_label` set but no claim file recorded get flagged as Signal 11 in sprint-audit output. This separates the audit concern from the workflow concern. Researcher is never blocked mid-campaign; the signal surfaces during sprint review. Trade-off: EPIC-6 errors are not caught before the conclusion verdict is written.
- [ai] Assumption reversal on Q3: What if claim files are placed in the project tree but under a path derived from campaign name (not campaign UUID) — `.bth/claims/<campaign-name>.claim.toml`? Campaign names are researcher-chosen strings (like "sp_0_6_nvt_sweep"), stable, human-readable, and exist before the campaign UUID is assigned. This enables convention-based auto-discovery at `bth conclude` without a `--claim` flag: bathos looks for `.bth/claims/<campaign-name>.claim.toml` automatically.
- [ai] Assumption reversal on Q1: What if the discriminability matrix is not in the claim file at all, but is instead declared in the campaign creation step — `bth campaign start --mode confirmation --hypotheses H1,H0_misspec --discriminates "H1 vs H0_misspec: discriminates"` — and stored in the campaigns table? Discriminability is a property of the campaign design, not the claim conclusion. This moves the check earlier (at campaign start) and treats it as part of the campaign pre-registration, not the claim authoring.
- [ai] Cross-cutting — migration plan: `bth migrate` adds four nullable columns to the DuckDB campaigns table: `claim_path TEXT`, `claim_sha256 TEXT`; adds two nullable columns to the runs cool-tier Parquet schema: `claim_discriminates TEXT` (comma-separated short IDs), `claim_isolates TEXT` (comma-separated short IDs). All nullable with NULL defaults. Existing campaigns and runs are unaffected. Schema version bumps to v7. This is strictly additive.
- [ai] Cross-cutting — adoption ladder enforcement: At MVP (Phase 1), Union Gate is opt-in per campaign: only campaigns that have a `claim_path` recorded in the catalog are subject to Union Gate check at `bth conclude`. Campaigns with `claim_path = NULL` proceed as before (no gate). This means the researcher adopts claim authoring voluntarily; the gate activates automatically once they author a claim. Phase 2 flips the default for confirmation-mode campaigns.
- [ai] Cross-cutting — `bth claim register` subcommand: links a claim file to a campaign without concluding. `bth claim register --campaign sp_0_6_nvt_sweep --claim .bth/claims/sp_0_6_nvt_sweep.claim.toml`. Writes `claim_path` and `claim_sha256` to campaigns table. Researcher can call this mid-campaign (after authoring the claim pre-registration) so that the file is locked to the campaign before results are seen. This prevents claim backdating — the SHA256 is recorded at registration time, not at conclude time.
- [ai] Assumption reversal — what if the Union Gate check is not at `bth conclude` but at `bth run` submission time? When the researcher submits a run for a confirmation-mode campaign that has a registered claim, bathos checks at submit time whether the run's sidecar `discriminates` field covers the hypotheses in the claim. If not, warns (or blocks). This pushes the check earlier — before the experiment runs — but requires the sidecar `discriminates` field to be populated before the run, which requires the claim to be registered first. Forms a natural pre-registration loop.
- [ai] Q5 cross-cut — `[baseline_parity]` block scope: the block lives in the benchmark sidecar (not the claim file) because baseline parity is a property of a specific benchmark run, not of a campaign conclusion. `check_baseline_parity` becomes a new Tier-2 lint check: queries warm catalog for `baseline_ref` run, retrieves its metric value, compares to current run's metric within `parity_tolerance`. If check passes, the run is admissible as a porting validation. If it fails, lint raises WARNING (not ERROR) because the researcher may intend to investigate the regression, not suppress it.

## Decision Log
- [ACCEPT] Q1-A: hand-authored TOML matrix with outcome-label vocabulary: [CRITIC] idea=Q1-A lens=edge-case finding=Matrix structure makes missing assertions visible as gaps; flat pair list (Q1-D) makes missed confounds structurally invisible — a researcher who omits a confounding pair produces a lint-passing but incomplete claim. severity=MAJOR. Condition to pass: the schema must treat blank matrix cells as UNASSERTED (lint ERROR) not as "discriminates=false" (silent pass). The discriminability matrix must be required to cover all hypothesis-pair × outcome-label combinations; partial matrices must fail claim_validate. This closes the quadratic-enumeration concern: with a well-scoped hypothesis slate (2-4 hypotheses, 2-3 outcome labels), the matrix is at most 4×3=12 assertions — tractable.
- [REJECT] Q1-D: flat pair list instead of matrix: [CRITIC] idea=Q1-D lens=edge-case finding=Flat pair list has an invisible-omission failure mode: confounding pairs the researcher does not enumerate are structurally indistinguishable from pairs the researcher assessed and found non-confounding. A lint cannot distinguish omission from affirmative assertion of non-confounding. severity=FATAL. The core purpose of the discriminability map is to prevent EPIC-6-class errors; a structure that makes confounds invisible defeats the purpose entirely.
- [ACCEPT] Q2-D: mode-gated escalation: [CRITIC] idea=Q2-D lens=second-order-effects finding=Mode-gated escalation has a perverse-incentive failure mode (mode=exploration to avoid confounded verdict) but this is a known design property of all bathos soft gates; the appropriate countermeasure is a new Sprint Audit Signal 12 flagging confirmation-intent campaigns with null claim_path, not a harder gate. severity=MAJOR. ## Assumptions

1. Single researcher using campaigns across 10+ projects; campaign `mode` is set once at creation and is authoritative — no mode changes after runs start.
2. Short IDs in `discriminates`/`isolates` are stable after claim registration; renaming hypothesis IDs after runs start is a breaking change requiring `--force` re-registration and an audit event.
3. `claim.bth.toml` is committed to git before any confirmatory run is submitted to SLURM — the file is authored pre-registration, not backdated.
4. Adoption-ladder: enforcement is warning-only for campaigns created before the feature ships; no retroactive enforcement on pre-existing campaigns.
5. TOST equivalence bounds in `[reference_parity]` are declared by the researcher; bathos enforces their presence and evaluates the result but does not set the bounds or choose the statistical procedure.
6. The `[discriminability]` matrix is authored before any confirmatory run; it may reference `planned_run_label` values that do not yet exist as actual run records.
7. `confounded` as a verdict is displayable by existing `rich_fmt` formatters — it is a free string in `outcome_label`, no enum change required.
8. `claim_discriminates` and `claim_isolates` stored as JSON array strings (e.g., `'["H_info_sym","H_topology"]'`) in cool-tier Parquet; DuckDB can query with `json_contains()` — avoids comma-separated substring collision (e.g., `H1` matching `H10`).

### AC-01 — bth claim scaffold
- Given a campaign ID, when `bth claim scaffold <campaign_id>` is called, then `.bth/claims/<campaign_name>.claim.toml` is created with all required sections: `[claim]` (headline, kill_condition, regime), `[[hypotheses]]` (≥2 entries, each with id, label, predicted_signature; one entry with id containing "null" or "misspec"), `[[assumptions]]`, `[[confounds]]`, `[[claim.discriminability]]` stub, `[union_gate]` with clauses list.
- Scaffold generates per-run sidecar snippets (printed to stdout or a sidecar_snippets/ subdir) with `discriminates` and `isolates` fields pre-populated from the hypothesis IDs in [[hypotheses]].

### AC-02 — bth claim register
- Given a valid claim.bth.toml path, when `bth claim register <path> --campaign <id>` is called, then `campaigns.claim_path` and `campaigns.claim_sha256` are written to the DuckDB campaigns table.
- Re-registering an already-registered claim requires `--force`; the re-registration writes an audit event with timestamp and reason.
- `claim_sha256` is computed at registration time (not at conclude time) — this is the tamper-evidence anchor.

### AC-03 — bth claim validate: required fields
- Given a claim.bth.toml file, when `bth claim validate <path>` is called:
  - Missing `headline` → ERROR
  - Blank or empty `kill_condition` → ERROR (uniquely mandatory; zero bypass justification)
  - Fewer than 2 `[[hypotheses]]` entries → ERROR
  - No `[[hypotheses]]` entry with id containing "null" or "misspec" → WARNING
  - Any `[[hypotheses]]` or `[[confounds]]` entry with `id` matching `/^[A-Z][0-9]+$/` and no `label` field, or blank `label` → WARNING
  - Blank matrix cell in `[[claim.discriminability]]` (asserted hypothesis-pair × outcome-label with no `verdict`) → ERROR

### AC-04 — discriminability zero-power lint
- Given a `[[claim.discriminability]]` block, when `bth claim validate` or `bth lint` is called, then any `planned_run_label` where all `(hypothesis_a, hypothesis_b)` pairs have the same `predicted_outcome` → WARNING: "zero discriminative power for run X — all hypothesis pairs predict identical outcome."

### AC-05 — positive-testing-bias lint
- Given `[[claim.discriminability]]` entries where every row predicts the primary hypothesis passing and all alternatives failing, when `bth claim validate` is called → WARNING: "positive-testing bias detected — no run in the matrix challenges the primary hypothesis."

### AC-06 — single-cell-gate lint
- Given a campaign with ≥2 confirmatory runs all sharing identical grid parameter values in their result_schema, when `bth lint` is called → WARNING: "single-cell-gate smell — all confirmatory runs use identical parameter values; claim.regime may not be covered."

### AC-07 — disconfirming-regime lint
- Given a claim with `regime` specifying a parameter range, when `bth lint` is called, then if the union of result_schema grid values across campaign runs does not span the stated regime → WARNING: "disconfirming regime excluded — claim.regime [X] not covered by any run."

### AC-08 — Union Gate at conclude
- Given a **confirmation/validation/production** campaign with a registered `claim_path`:
  - Any clause in `[union_gate.clauses]` with no run having that clause_id in its `discriminates` field and non-null outcome → verdict downgraded to `confounded` (not `pass`)
  - Two runs mapping to the same clause with contradictory outcomes (one pass, one fail) → WARNING: "contradicted clause C1 — runs R1 and R2 disagree" (not blocked at MVP; hard-block post-MVP)
- Given an **exploration/calibration** campaign: same checks run but → WARNING only, no downgrade
- Given ANY campaign with `claim_path IS NULL`: Union Gate short-circuits (no check, no message)

### AC-09 — Union Gate bypass
- Given a confirmation campaign with unmapped clauses, when `bth campaign conclude --outcome pass --force-verdict` is called: conclude proceeds, `claim_mode='bypassed'` written to campaigns table, warning printed: "Union Gate bypassed — unmapped clauses: [C1, C3]."

### AC-10 — Signal 12
- Given a confirmation campaign with null `claim_path`, when `bth sprint-audit` is called → Signal 12 fires: "Confirmation campaign <name> has no registered claim — union gate will not fire at conclude."
- Signal 12 ships in the same PR as the Union Gate (co-ship constraint).

### AC-11 — claim_sha256 integrity at conclude
- Given a claim.bth.toml that has been modified after `bth claim register`, when `bth campaign conclude` is called: `SHA256(current claim file) ≠ campaigns.claim_sha256` → ERROR: "claim.bth.toml has been modified since registration — re-register with `bth claim register --force` to acknowledge the amendment."

### AC-12 — claim-coverage report
- Given a campaign conclude that passes (or bypasses) the union gate, when conclude completes → `claim_coverage_<campaign_id>.json` emitted at `~/.bth/catalog/sidecars/<campaign_id>/` with fields: `coverage_fraction`, `uncovered_clauses[]`, `contradicted_clauses[]`, `covered_clauses[]`, `verdict_blocked` (bool), `bypass_reason` (str or null).

### AC-13 — [baseline_parity] in confound register
- Given a `[[confounds]]` entry with a `[confounds.reference_parity]` sub-block (fields: `reference_paper`, `reference_metric`, `reference_value`, `equivalence_bound`, `parity_run_id`):
  - `parity_run_id` empty → WARNING: "baseline admissibility not established for <confound_label>"
  - `parity_run_id` set but `reference_metric` key not found in run metadata JSON → ERROR: "parity_metric key not found in baseline run metadata — check field name"
  - `parity_run_id` set, metric found, `|result - reference_value| < equivalence_bound` → PASS (confound status transitions to `controlled`)
  - `|result - reference_value| >= equivalence_bound` → WARNING: "parity run <id> does not satisfy equivalence bound"

### AC-14 — descriptive labels in human-facing output
- Given any lint warning, conclude-time feedback, or claim-coverage report that references a hypothesis or confound, the message uses the `label` field (not `id`) in all human-facing text; machine cross-references (TOML fields, DuckDB queries) use `id`.

### AC-15 — §10.3 skill update
- Given the `/using-bathos` skill, when updated, a "Signal discrimination and probe design" section is present covering probe vocabulary: scaled-divergence probes, planted-mode probes, null-injection probes, information-ablation probes — each with expected signature and what it discriminates.
